# Entity hunks: one list sorted by anchor, with an anchor-definition-site fallback snippet

## Context

`groupRdfDiffByEntity` (the grouping behind `diff -f html` and the webapp diff page) buckets changed quads into **entity hunks** keyed by an **anchor** — a named entity's IRI, or the root canonical bnode label of an orphan blank-node tree. It returns `HunkedRdfDiff { changed, removed, added, totals }`: three arrays, one per `state`, each independently sorted lexicographically by anchor. The renderers concatenate them `changed → added → removed` — the in-code rationale was "the changed-on-both-sides case is the most informative, so render it first." The result is that a hunk's position on the page is a function of *what kind of change it is* before *which subject it is*: two hunks for adjacent subjects (`ex:Alice`, `ex:Bob`) land in different parts of the page if one was edited and the other newly added, and scanning for "what happened to subject X" means looking in up to three places.

Separately: a `changed` hunk renders a left snippet panel and a right snippet panel, each fed from **Source records** gathered off that hunk's *changed* lines. When a property is *added* to an already-existing subject, the left side contributed no changed lines, so its panel shows `(absent on left)` — even though the subject plainly exists on the left. The placeholder reads as "this subject is new," which is wrong; the reader gets no pointer to where the subject actually lives in the left file.

## Decision

**Entity hunks are presented as a single list sorted purely by anchor.** `HunkedRdfDiff` becomes `{ hunks: Hunk[], totals }`. `groupRdfDiffByEntity` emits one array sorted by a single comparator: lexicographic on the anchor string, with `state` as the only tie-break (`removed` < `changed` < `added`) — the tie-break exists solely to give a deterministic order to a left-only and a right-only **orphan hunk** that happen to share a canonical bnode label. Orphan hunks (anchors rendered `_:label`) sort naturally by that string and are not carved out to a separate region. Every `Hunk` already carries `state`; it now drives only the hunk's accent colour, not its position. Within a hunk, line order is unchanged (subject path → predicate → side → object via `compareHunkLines`).

**A `changed` hunk carries its anchor's definition site on any side where it exists but contributed no changed-line source records.** New field `Hunk.anchorSource?: { left: SourceRecord[]; right: SourceRecord[] }`. For a `changed` hunk, `anchorSource.left` is populated only when (a) the left contributed zero changed-line source records *and* (b) the anchor IRI is present in the left store — and then it holds one **Source record** per file the anchor's triples are annotated from, each focused on the earliest annotated line of the anchor in that file. Symmetric for `right`. For `added` / `removed` hunks, and whenever the anchor is genuinely absent on a side, the field stays empty.

**Renderers show the definition site under a muted `defined here` heading; `(absent …)` is reserved for true absence.** In a hunk's snippet area, a side shows: its changed-line **Source-file snippets** when it has any; otherwise its `anchorSource` snippets under a subdued `defined here` label (visually distinct from the change-coloured `left` / `right` headers, so it is not read as a change); otherwise the `(absent on left/right)` placeholder. This applies uniformly to the webapp diff page and the CLI `html` format — no flag toggles either behaviour.

## Considered alternatives

- **Keep the three `state` arrays; sort within and concatenate `changed → added → removed`.** Status quo. Rejected: that ordering *is* the thing being changed. The buckets only exist to impose it; once hunks sort by anchor, the arrays are dead weight and the data structure would permanently misrepresent intent. `state` is already per-hunk, so nothing is lost by flattening.
- **Keep the three arrays but merge-sort by anchor at render time.** Rejected for the same reason — the model would still advertise a grouping no consumer wants, and two renderers (webapp, CLI html) would each have to re-merge. Flattening once in `groupRdfDiffByEntity` is the single point of truth.
- **Pin orphan hunks to the end of the list.** Rejected: "sorted purely by anchor" is most honestly one comparator with no regions. Orphan hunks are rare; letting `_:label` anchors fall where the string sorts is predictable and needs no carve-out. (Revisit only if orphan-heavy diffs prove confusing in practice.)
- **Just append the anchor's definition record into `hunk.sourceRecords.left` when that side is empty.** Rejected: that conflates "this is what changed here" with "this is where the subject lives" — the snippet would render identically to a real change snippet with no signal that it isn't one. A separate `anchorSource` field plus a `defined here` heading keeps the distinction visible.
- **Show *all* of the anchor's existing source records on the empty side, not just the first line per file.** Rejected: that can be a large block for a many-property subject, and the goal is a *pointer* ("here's where this thing is"), not a re-display of the unchanged definition. The earliest annotated line per file is the literal "where it's first encountered" and stays bounded — usually one snippet.
- **Privilege the `rdf:type` triple's line as the definition site.** Rejected: not every subject is typed, and "first encountered" — the minimum annotated line — needs no such assumption and degrades gracefully.
- **Compute the definition-site lookup in the webapp.** Rejected: the webapp only has `hunk.sourceRecords`; the side stores (with their RDF-star `sparqly:source` annotations) live in `libs/core` at grouping time. The backend is the only place that can resolve "where is anchor X in the left file," so it surfaces `anchorSource`; the renderers stay presentational.

## Consequences

- **`libs/core`**: `HunkedRdfDiff` is `{ hunks: Hunk[], totals }` — the `changed` / `removed` / `added` arrays are gone. `groupRdfDiffByEntity` emits one anchor-sorted, `state`-tie-broken list and, per `changed` hunk, fills `Hunk.anchorSource` from each side's store when that side has no changed-line source records but does contain the anchor. `Hunk.state` is retained (accent colour only); its doc-comment no longer references "sections." `compareHunkLines` is untouched.
- **CLI `html` format**: the composer iterates the single `hunks` list (no concat); a side with no changed-line snippets renders the `anchorSource` snippets under a `defined here` heading, falling back to the `(absent …)` text only when both are empty.
- **Webapp**: `diff-result-renderer.component.ts` reads `hunked.hunks` directly (the `[...changed, ...added, ...removed]` concat is removed); `diff-hunk.component.ts` derives `defined here` snippet ranges from `hunk.anchorSource` and shows the `(absent on left/right)` placeholder only when a side has neither changed-line ranges nor an `anchorSource`.
- **Wire shape**: the grouped diff response (`GET /api/diff` and the `json`-adjacent grouped payload) carries `hunks: [...]` instead of `{ changed, removed, added }`, plus `anchorSource` on `changed` hunks where applicable. Pre-1.0; the only client is the webapp, updated in lockstep.
- **CONTEXT.md**: a new **Entity hunk** entry records the anchor, orphan hunks, the single-sorted-list ordering rule, the `state`-as-colour-only role, and the anchor-definition-site fallback snippet.
- **Tests**: `group-rdf-diff-by-entity.spec.ts` / `.golden.spec.ts` move from asserting per-section bucketing to asserting one anchor-sorted list and `anchorSource` population; the webapp diff-page specs assert the single list and the `defined here` snippet for an added-property-on-existing-subject hunk (the `ex:address` block now in `test/data/diffs/diff-01.ttl` is a fixture for exactly that case).
