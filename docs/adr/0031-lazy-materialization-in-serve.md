# Lazy materialization in `serve`

Eager boot-time resolution of every materialized source in `serve` is paying full I/O + parse + transform cost for sources nobody queries in this session — and `expandSplitGlobs` already takes long enough at startup for config-heavy registries that the latency is user-visible. We're moving the in-memory Store build off the boot path: `serve` now builds each source's Store on the first request that touches its engine and memoizes it for the life of the process. The registry, its split-glob children, and the snippet allow-list's glob-path enumeration stay eager so `/api/config` and shared-link snippet requests still work before any source is touched. Single-target CLI commands (`query`, `hash`, `diff`) are unaffected — their target is what was asked for, so loading it is the work the command was invoked to do.

This supersedes the "**materialized** sources resolve eagerly at boot" sentence in ADR-0011 §Registry mode.

## Considered options

- **Per-source `eager: true` opt-in.** Rejected for scope-1: adds schema/docs/test surface for a hypothetical use case (boot-time fail-fast for ops alerting). The retry-on-rejection error-handling contract below already gives a fail-soft story that's *better* than eager boot's fail-loud crash, weakening the strongest argument for the flag. Adding the flag later is non-breaking.
- **Global `serve --preload` flag.** Same rejection rationale as above.
- **Background prefetch after `app.listen`.** Plausible alternative form of "lazy". Rejected as the default because it still pays the boot cost — just out of the HTTP listen path — which forfeits the actual win for config-heavy registries on contributor machines and CI smoke checks. Same `--preload`-style flag could add it back later.
- **Push async into `QueryEngine` itself** (accept `() => Promise<Store>`). Rejected: spreads the lazy contract across `core` and all of its existing call sites and specs. Hoisting the async into `EngineMap.ensure(id): Promise<QueryEngine>` localizes the change to the server boundary; `QueryEngine` stays synchronous.
- **Sticky failure (cache rejection forever).** Rejected: the `--watch` system already exists so file fixes don't require a restart — sticky failure would contradict that posture. Instead we clear the memoized promise on rejection; concurrent requests share one in-flight attempt (no thundering herd within an attempt), and the next request after rejection retries fresh.
- **Lazy snippet allow-list (grow as sources load).** Rejected: bookmarked/shared snippet links would 403 until a prior request happened to load the underlying source, and the snippet endpoint would need a path→source reverse-lookup it doesn't have today. Instead we lift glob-path enumeration into a separate phase that runs eagerly (cheap FS walk, no parsing) so the allow-list is fully populated at boot — matching the same "enumerate eager, parse lazy" split that `splitByFile: true` already uses.

## Consequences

- `EngineMap.get(id)` becomes `EngineMap.ensure(id): Promise<QueryEngine>`. Each `Entry` holds a memoized `Promise<LoadedEntry>` that's `undefined` until first `ensure`; concurrent first-touches share the in-flight promise. Controllers `await ensure` before using an engine.
- Load failure clears the memoized promise (next request retries). Load errors are routed through `resolveSourceResult` (Result-typed; ADR-0024) rather than the deprecated throwing `resolveSource` adapter, so failures surface as proper HTTP status (4xx for user-input errors like a bad `@id:ref`) instead of 500s. This is a small bonus migration off `@deprecated resolveSource` at the same call site.
- `SnippetAllowList` is populated at boot from a new "enumerate glob paths without parsing" core helper, applied to *every* glob (split or not). Eager `expandSplitGlobs` already does this for split-glob children; the new helper extends it to plain globs.
- `MetaChildrenCache` keeps updating on FS events for un-touched sources (registry listing depends on it). Engine rebuilds on FS events skip un-touched sources — `if (!entry.loaded) return;` — so the watcher does not subvert laziness.
- The `serve-ready` boundary log line still reports `sources: N` (the count of served entries) but no longer implies "N Stores in memory." Per-source load timing surfaces on first request via the existing `source-loaded` debug event, just deferred from boot.
- Floating `gitRef` globs no longer snapshot their resolved SHA at server boot — the snapshot is taken on first touch. Not a regression vs the floating-vs-pinned contract in ADR-0029 (which never promised boot-time pinning), but worth naming: long-running `serve` processes with floating refs will see "first request decides the SHA for the life of the process," same as eager today, just with the decision point shifted later.
