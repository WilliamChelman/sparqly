# Typed error handling with `neverthrow`

We want type-safe error handling so functions advertise what they can fail with, and so expected domain errors reach the UI as structured information instead of leaking through as unclassified 500s (the trigger was tabular diff with a blank-node-valued column producing a generic 500 — the throw at `tabularRowKey` had no path to the existing in-band error envelope `runTabular` builds for shape errors). We adopt `neverthrow` and convert `throw` → `Result` across `libs/core` and `libs/server`, leaving `apps/web` and `apps/cli` as consumers of the resulting Results.

## Decision

- **Library: `neverthrow`.** `ResultAsync<T, E>` for async (the dominant case — Comunica execution, fs IO, fetch); `Result<T, E>` for sync. Linear-with-early-exit pipelines (e.g. `runGraph`, `runTabular`, `resolveGraphSide`) use the `safeTry` generator for sequencing rather than chained `.andThen`.
- **Per-module tagged-union errors.** Each feature folder owns its error type — e.g. `libs/core/diff/errors.ts` declares `type DiffError = TabularBlankNodeError | EndpointAsDiffTargetError | MixedShapeError | …`. Cross-folder calls compose via union (`Result<T, DiffError | SourceError>`). No global `AppError` union — it would couple every feature to every other and grow unboundedly.
- **Variants carry structured data; a `format*Error(e): string` next to the type definition produces the human string.** Single source of truth for wording across the web UI, the CLI, and stderr/log lines. Structured fields stay queryable for UI surfaces that want them (e.g. highlighting the offending `?column` for `tabular-blank-node`).
- **Pragmatic throw policy.** Domain errors and wrapped third-party calls (Comunica, n3, fs, fetch) go through `Result`. "This can't happen" invariants — e.g. `engine-map.ts` failing to find a source `@id` that the registry-lookup precondition guarantees exists — stay as `throw`. They are bugs, not states; surfacing them as `Result` variants would force every `match()` to handle a never-happens branch and tempt callers to `.unwrap()` (which throws — defeating the point). Top-level Nest exception filter catches stray throws as 500s with stack.
- **In-band error envelope preserved for expected user-input errors.** The `diff` service today returns `200 + { kind: 'error', errors: { left?, right?, top? } }` for shape errors so the web UI can render them inline next to the offending side. We keep this envelope; the new error variants (including `tabular-blank-node`) flow into it. HTTP 4xx/5xx remain reserved for transport-level failures (malformed JSON, unknown source `@id`, upstream endpoint unreachable). Switching expected errors to HTTP statuses would break the inline-error UX with no offsetting benefit.
- **Result → HTTP at the controller, per route.** Each controller method does `result.match(ok, err => …)` and either returns the in-band envelope or throws an `HttpException` via a per-feature error mapper module (`libs/server/src/lib/diff/diff-http-errors.ts`). No global Nest exception filter mapping domain errors — that would couple top-level `libs/server` to every feature folder.
- **CLI shares the formatter, decorates around it.** `apps/cli` calls the same `format*Error(e)` the server uses; ANSI/colors and exit-code mapping are CLI-only concerns layered on top. Wording cannot drift between the two surfaces.
- **Cross-feature composition: wrap, don't duplicate.** When feature `A`'s pipeline calls into feature `B`, `AError` carries the upstream as a wrapper variant (`SourceWrappedError { side?, source: SourceError }`, `TargetWrappedError { side?, target: TargetError }`) rather than re-declaring `B`'s variants in `A`'s union. Per-call context (`side: 'left' | 'right'`) lives on the wrapper, not the wrapped — `SourceError`/`TargetError` know nothing about diff sides. The `format*Error` delegates: `case 'source': return formatSourceError(e.source);`. Wording stays single-source-of-truth, and `A` doesn't have to re-version when `B` adds a variant. Corollary: a feature folder with **no variants of its own** doesn't get a union at all — its CLI/controller consumes upstream `Result<T, BError | CError>` directly. Adding a union later, once the feature owns its first variant, is a non-breaking change for `match()` callers.

## Migration

- **Vertical slice per feature folder.** `diff` first — biggest beneficiary (the blank-node bug), already half-shaped by the homegrown `*Safe` + `{ kind: 'ok' | 'err' }` pattern in `diff.service.ts`, and a contained surface to debug the convention on. Subsequent slices: `sources`/`target`, then `query`, `hash`, `format`, `serve`, `describe`.
- **Coexistence rule for shared core leaves: parallel-name primary impl.** When a slice needs to convert a leaf shared with non-converted callers (e.g. `resolveSource`, called from `diff`, `describe`, `bootstrap`, `engine-map`), introduce `resolveSourceResult` as the new primary `ResultAsync` implementation; rewrite the existing `resolveSource` as a thin throw-wrapping adapter around it and mark `@deprecated`. Non-converted callers stay on the legacy name, switching at their own pace. The legacy adapter is deleted when the last `@deprecated` import is gone. Avoids the all-call-sites-in-one-PR sprawl of pull-in, and the maintenance burden of two parallel implementations.
- **Zod adapter.** `.safeParse()`'s `{ success, error }` shape adapts into `Result` at the call site (`Result.fromSafeParseResult` helper or inline ternary). No new validation throws to convert.

## Considered alternatives

- **Targeted fix only** — define a `DomainError` class hierarchy + Nest exception filter mapping to 4xx, fix the blank-node throw site, do not adopt `neverthrow`. Rejected: solves the UI bleed but leaves the stated type-safety motivation unaddressed; throws remain unannounced in signatures.
- **Global `AppError` discriminated union (one type, every variant).** Rejected: every feature change ripples through every consumer; the union becomes a coupling sink at ~50+ variants.
- **Error subclasses (`class TabularBlankNodeError extends DomainError`).** Rejected: reintroduces the `instanceof`/`Error` runtime machinery this work is trying to escape; loses TypeScript exhaustiveness checks in `match()`.
- **Strict throw policy (every throw becomes a `Result` variant, including invariants).** Rejected per above — forces "impossible" branches through every `match`, encourages `.unwrap()` (which is a throw in disguise).
- **`Promise<Result<T, E>>` instead of `ResultAsync`.** Rejected: chained operations require `await` + `isErr()` check at each step, exactly the boilerplate `ResultAsync` + `safeTry` removes.
- **Switch all expected errors to HTTP 4xx with structured body** instead of preserving the in-band envelope. Rejected: would force a web-UI rewrite of inline error rendering with no observable user benefit.
- **Pull-in coexistence (slice converts every leaf it touches).** Rejected: diff's first slice would balloon to include `describe`, `bootstrap`, `engine-map` — stops being a slice.
